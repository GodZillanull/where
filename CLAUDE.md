# Where - 週末おでかけ提案アプリ

## コンセプト

このアプリは以下の課題を解決する：

- 平日夜や休日に「何かしたい」と思うが具体的な行き先が思い浮かばない
- SNSで情報収集するが情報が多すぎて結局決められない
- 結果的に家にいることが多く、新しい体験ができずにいる

## デザイン原則

### 1. 決断を後押しする
- 選択肢を絞る（3つ程度）
- 「これでいい」ではなく「これがいい」と思わせる
- 迷わせない、考えさせない

### 2. 行動を促す
- 「今週末」「今日」など時間的な緊急性
- CTAボタンは目立たせる
- 具体的な次のアクションを明示

### 3. ワクワク感を与える
- 色で感情を動かす（無彩色だけにしない）
- キャッチコピーで期待を高める
- 「行ったらこうなる」イメージを持たせる

### 4. 情報設計（深津流）
- 最初に「勝ち筋」を見せる
- 段階的開示（全部見せない）
- ユーザーの文脈に寄り添う

### 5. 視覚設計（catnose流）
- 余白を大切に
- 1画面1メッセージ
- ノイズを減らす（でも感情は動かす）

## ターゲットユーザー

- 20-30代
- 「何かしたい」けど腰が重い
- 情報過多で決められない
- 新しい体験への欲求はある

## 技術スタック

- React + Vite
- Tailwind CSS
- モバイルファースト

## Git ワークフロー

**重要: コード変更前に必ず以下を実行すること**

1. **ブランチを切る**（1つの変更につき1ブランチ）
   ```bash
   git checkout -b feature/機能名
   # または
   git checkout -b fix/修正内容
   ```

2. **変更を加える**

3. **コミット & プッシュ**
   ```bash
   git add .
   git commit -m "変更内容の説明"
   git push -u origin ブランチ名
   ```

4. **PRを作成して送信**
   - 変更が完了したら必ずPRを作成する
   - PRのURLをユーザーに共有する

**禁止事項:**
- mainブランチへの直接コミット
- PRなしでのマージ
- 1つのブランチに複数の無関係な変更を混ぜる
